# Vibe Dating MVP Feature Spec

작성일: 2026-06-14

## 1. 문제정의

커플은 주말 데이트를 계획할 때 괜찮은 장소를 찾기 위해 네이버 지도에서 직접 장소를 검색하고, 플레이스 페이지를 하나씩 열어보며 비교한다.

하지만 실제로 마음에 드는 장소를 찾기까지 여러 장소를 반복해서 확인해야 하므로 시간과 공수가 많이 든다.

Vibe Dating이 해결하려는 핵심 문제는 데이트 장소 탐색 과정에서 사용자가 직접 해야 하는 수고로운 선별 작업을 대신해주는 것이다.

## 2. 타깃 사용자와 사용 상황

### 타깃 사용자

Vibe Dating의 초기 타깃은 특정 연령, 연애 기간, 직업군으로 제한하지 않는다.

핵심 타깃은 데이트코스를 짜야 하는 모든 커플, 특히 데이트 전 장소를 직접 찾아보고 비교하는 사람들이다.

데이트 장소 탐색은 커플 두 사람이 똑같이 나눠서 하기보다, 한 사람이 더 자주 맡는 경우가 많다. 따라서 MVP는 둘이 같이 쓰는 앱이기 이전에, 데이트 계획을 주도하는 한 사람이 빠르게 좋은 후보를 찾는 도구로 출발한다.

### 사용 상황

가장 중요한 사용 상황은 주말 데이트 전날 계획을 짤 때다.

사용자는 다음 날 갈 만한 곳을 찾기 위해 네이버 지도에서 여러 장소를 검색하고, 플레이스 페이지를 하나씩 확인하며 괜찮은 곳을 고른다.

Vibe Dating은 이 과정에서 특정 지역 안의 데이트 가능한 장소들을 미리 선별해 보여줌으로써, 사용자가 직접 검색하고 비교하는 공수를 줄여준다.

### 초기 지역 범위

MVP는 처음부터 모든 지역을 열지 않는다.

우선은 사람들이 많이 찾는 유명 데이트 상권 또는 핫플 지역 일부만 오픈한다. 서비스 반응이 좋으면 이후 다른 지역으로 확장한다.

### 장소 카테고리

장소 카테고리는 음식점이나 카페로만 제한하지 않는다.

데이트가 가능한 모든 장소를 포함 대상으로 본다.

MVP에서 우선 사용하는 1-depth 장소 카테고리는 다음과 같다.

- 식사
- 카페·디저트
- 문화·관람
- 체험·액티비티
- 산책·자연·야외
- 쇼핑·구경
- 술·야간
- 휴식·힐링

구체적인 중분류 카테고리는 [장소-카테고리.md](장소-카테고리.md)를 기준으로 한다.

MVP에서는 선정한 지역 안에서 데이트 장소로 추천할 만한 후보를 최대한 폭넓게 포함한다.

## 3. 핵심 가치와 제품 가설

### 핵심 가치

Vibe Dating의 핵심 가치는 데이트 장소와 코스를 찾고, 원하는 분위기에 맞게 수정하는 시간을 줄여주는 것이다.

사용자는 네이버 지도에서 장소를 하나씩 검색하고 플레이스 페이지를 확인하면서 괜찮은 곳을 직접 골라야 한다. 장소를 고른 뒤에도 이동 거리, 분위기, 카테고리, 순서를 다시 맞추기 위해 여러 후보를 반복해서 바꿔보는 경우가 많다.

Vibe Dating은 이 수고로운 탐색과 수정 과정을 대신해, 사용자의 성향에 맞는 장소와 경로 후보를 먼저 추천하고, 사용자가 자연어로 데이트 코스를 수정할 수 있게 한다.

사용자가 서비스 사용 후 느껴야 하는 가치는 다음과 같다.

- 일일이 찾지 않아도 괜찮은 후보가 바로 나왔다.
- 내가 원하는 분위기의 장소만 잘 추천받았다.
- 말로 요청하듯이 데이트 코스를 쉽게 바꿀 수 있었다.
- 데이트 계획 짜는 게 훨씬 편해졌다.

### 네이버 지도와의 차이

네이버 지도, 카카오맵, Google Maps는 플레이스의 상세 정보를 제공하는 서비스다.

반면 Vibe Dating은 플레이스 자체를 보유하거나 지도 서비스를 대체하려는 서비스가 아니라, 여러 플레이스 중 사용자의 성향에 맞는 장소와 경로를 추천해주는 서비스다.

구체적인 플레이스 정보는 네이버, 카카오, Google 같은 외부 지도/플레이스 서비스로 연결한다. Vibe Dating은 그 전에 사용자가 어떤 장소를 볼지, 어떤 경로를 고려할지 좁혀주는 역할을 한다.

### 추천 원칙

추천 결과는 절대적인 장소 점수로 정하지 않는다.

어떤 장소가 모든 사람에게 좋은 장소라고 판단하는 방식이 아니라, 사용자 성향에 따라 상대적으로 더 잘 맞는 장소를 점수화한다.

같은 장소라도 조용한 대화 중심 데이트를 원하는 사용자에게는 높은 점수를 받을 수 있고, 활동적이고 새로운 체험을 원하는 사용자에게는 낮은 점수를 받을 수 있다.

### 핵심 제품 가설

사람들은 데이트 장소를 찾기 싫어하는 것이 아니라, 괜찮은 곳을 찾기까지 반복 검색하고 비교하는 과정이 수고롭다고 느낀다.

사용자의 성향에 맞는 장소와 경로를 먼저 추천해주고, 이를 바이브에 맞게 자연어로 수정할 수 있게 하면 데이트 계획의 부담이 크게 줄어든다.

### 차별 기능: 사용자 성향 기반 추천

사용자 성향 기반 추천은 회원가입 시 저장된 기본 데이트 성향을 바탕으로, 현재 지도 영역 안의 플레이스를 사용자에게 잘 맞는 순서로 정렬해 보여주는 기능이다.

Vibe Dating은 단순히 가까운 장소나 인기 장소를 나열하지 않고, 사용자가 선택한 성향 문장과 플레이스별 성향 점수를 조합해 추천 점수를 계산한다.

예시:

- 조용한 곳을 선호하는 사용자에게는 대화하기 좋은 장소를 더 높게 보여준다.
- 사진 찍는 걸 좋아하는 사용자에게는 포토 스팟이나 분위기 좋은 장소를 더 높게 보여준다.
- 실내 위주 데이트를 선호하는 사용자에게는 날씨 영향을 덜 받는 장소를 더 높게 보여준다.

### 차별 기능: 바이브 데이트 코스 수정

바이브 데이트 코스 수정은 추천된 장소 후보를 데이트 코스로 묶거나, 생성된 코스를 사용자가 대화형으로 조정하는 기능이다.

예시:

- 조금 더 조용한 곳 위주로 바꿔줘.
- 카페는 빼고 술집을 넣어줘.
- 이동 적은 코스로 바꿔줘.
- 좀 더 트렌디한 분위기로 추천해줘.
- 비 오는 날 가기 좋은 실내 위주로 바꿔줘.

Vibe Dating은 지도 위에 사용자 성향 기반으로 스코어링된 플레이스와 경로를 추천하고, 사용자가 대화형으로 데이트 코스를 바이브에 맞게 수정할 수 있는 데이트 코스 추천 서비스다.

## 4. MVP 목표와 기능 범위

### MVP 검증 목표

MVP에서 검증해야 하는 핵심은 다음 세 가지다.

1. 성향 기반 장소 추천이 실제로 유용한가
2. 웹 상에 있는 데이트 장소 관련 소스를 가져와 AI 분석으로 신뢰할 만한 점수 측정이 되는가
3. 바이브 데이트 코스 수정이 좋은 사용자 경험을 만드는가

### 데이트 장소 추천 범위

MVP의 기본 출발점은 장소 추천이다.

사용자는 지도 화면에서 현재 지도 영역 안의 데이트 가능 플레이스를 조회하고, 저장된 사용자 성향을 기준으로 추천 점수순으로 정렬된 장소 후보를 확인한다.

장소 추천은 MVP의 기본 경험이며, 처음부터 자동으로 데이트 코스를 만들어주는 것이 아니라 먼저 사용자 성향에 맞는 플레이스들을 추천한다.

### 플레이스 데이터 수집 및 AI 점수화 어드민 범위

MVP에는 추천 품질 검증을 위해 플레이스 데이터를 수집하고 AI로 점수화하는 최소 어드민 페이지와 기능을 포함한다.

운영자는 어드민에서 플레이스별 웹 크롤링 또는 웹서치를 실행하고, 수집된 출처 링크와 요약 정보를 확인할 수 있어야 한다.

운영자는 수집된 정보를 기반으로 AI 분석을 실행해 플레이스별 성향 점수와 요약을 생성할 수 있어야 한다.

어드민 기능의 목적은 완성된 운영 시스템을 만드는 것이 아니라, MVP 추천 결과의 기반이 되는 Place DB와 태그별 점수 데이터를 빠르게 만들고 검증하는 것이다.

### 바이브 데이트 코스 수정 범위

바이브 데이트 코스 수정은 MVP에 포함한다.

바이브 데이트 코스 수정은 사용자가 추천된 플레이스를 코스로 만들거나, 생성된 코스를 대화형으로 수정하는 기능이다.

코스 생성도 바이브 데이트 코스 수정의 한 형태로 본다. MVP에서는 자동으로 상시 코스를 생성하지 않고, 사용자가 자연어로 요청했을 때 추천된 플레이스나 선택된 플레이스를 기반으로 코스를 생성하거나 수정한다.

예시:

- 여기에서 제일 괜찮은 코스 하나만 만들어줘.
- 이 장소들이랑 이어서 코스 만들어줘.
- 이동 적은 순서로 묶어줘.
- 두 번째 카페는 마음에 안 들어. 다른 걸로 넣어줘.
- 저녁 파스타 말고 고기.
- 조금 더 조용한 곳 위주로 바꿔줘.
- 이동이 너무 길어. 가까운 곳끼리 묶어줘.
- 카페 하나 빼고 전시를 넣어줘.

MVP에서 바이브 데이트 코스 수정은 추천된 플레이스들을 기반으로 장소 추가, 제거, 교체, 재조합, 코스 생성, 순서 변경, 경로 수정이 가능한 핵심 대화형 편집 기능으로 정의한다.

## 5. 핵심 사용자 플로우

1. 사용자가 앱에 진입한다.
2. 회원가입 또는 로그인을 한다.
3. 회원가입 시 기본 데이트 성향 문장을 선택한다.
4. 사용자가 지도 화면에 진입한다.
5. 현재 지도에 보이는 영역을 기준으로 데이트 가능 플레이스가 조회된다.
6. 저장된 사용자 성향을 기반으로 플레이스별 추천 점수가 계산된다.
7. 플레이스는 추천 점수 순서로 리스트에 표시된다.
8. 지도 위 마커는 추천 점수에 따라 시각적 강약이 다르게 표시된다.
9. 사용자는 지도와 리스트를 함께 보며 추천 후보를 확인한다.
10. 각 플레이스 카드에서 간단 요약, 선호 항목별 점수, 외부 서비스 평점, 외부 링크를 확인한다.
11. 사용자는 채팅창에서 바로 자연어 요청을 입력한다.
12. 서비스는 요청에 따라 추천 결과를 수정하거나 코스를 생성한다.
13. 생성된 코스 또는 수정된 추천 결과가 지도와 리스트에 다시 표시된다.
14. 사용자는 최종적으로 마음에 드는 플레이스의 외부 지도 링크로 이동해 상세 정보를 확인한다.

## 6. 도메인 모델

### 사용자

사용자는 회원가입과 로그인을 통해 기본 데이트 성향을 저장한다.

MVP에서는 커플 계정이 아니라 개인 계정을 사용한다.

### 기본 성향

기본 성향은 사용자가 회원가입 시 선택하는 장기 취향이다.

성향은 단어형 태그가 아니라 문장형 선택지로 제공한다.

구체적인 성향 문장과 내부 태그 체계는 [사용자-성향.md](사용자-성향.md)를 기준으로 한다.

예시:

- 조용한 곳이 좋아요.
- 사진 찍는 걸 좋아해요.
- 대화하기 좋은 곳이 좋아요.
- 걷기 좋은 데이트가 좋아요.
- 실내 위주가 좋아요.
- 트렌디한 곳이 좋아요.

초기에는 데이트 성향을 표현할 수 있는 문장 몇 개만 넣고, 이후 사용자 반응을 보면서 다듬는다.

사용자는 성향 문장을 제한 없이 여러 개 선택할 수 있다.

### 플레이스

플레이스는 데이트 장소의 기본 단위다.

음식점, 카페, 술집, 전시, 팝업, 공방, 산책 장소 등 데이트가 가능한 모든 장소가 포함될 수 있다.

플레이스는 서비스 안에서 유니크하게 관리되어야 한다.

Vibe Dating은 네이버 지도 기반으로 동작하므로, 각 플레이스는 네이버 지도 API에서 제공되는 플레이스 ID와 1:1로 매핑되어야 한다.

### 플레이스별 성향 점수

각 플레이스는 여러 성향 문장 또는 성향 태그에 대해 점수를 가진다.

플레이스 관련 정보는 웹 크롤링, 웹서치, 외부 데이터 등을 통해 수집하고, AI가 이 정보를 분석해 태그별 점수를 매긴다.

최종적으로는 플레이스마다 태그별 점수가 저장된다.

예시:

- 조용한 곳이 좋아요: 85점
- 사진 찍는 걸 좋아해요: 72점
- 트렌디한 곳이 좋아요: 64점

점수 산정 방식과 데이터 수집 방식은 별도 기술 검토 문서에서 다룬다.

### 사용자 성향 기반 추천 점수

사용자가 선택한 성향 문장들을 기준으로 각 플레이스가 얼마나 잘 맞는지 계산한다.

플레이스 자체에 절대 점수를 매기는 것이 아니라, 사용자 성향과 플레이스별 성향 점수를 조합해 추천 점수를 만든다.

### 선호 항목별 점수 표시

플레이스 카드에는 별도의 설명 문장을 생성해 보여주기보다, 사용자가 선택한 성향에 대응하는 항목별 점수를 보여준다.

각 항목별 점수는 플레이스별 성향 점수를 기반으로 표시한다.

최종 추천 점수는 사용자가 선택한 성향 항목들의 점수를 특정 알고리즘으로 합산해 계산한다.

예시:

- 조용한 곳이 좋아요: 85점
- 사진 찍는 걸 좋아해요: 72점
- 실내 위주가 좋아요: 61점

사용자는 자신이 선호한 항목별 점수를 보고 왜 이 장소가 상위에 노출되었는지 판단한다.

### 코스

코스는 여러 플레이스의 조합이다.

처음부터 자동으로 항상 생성되는 것은 아니고, 사용자가 요청했을 때 생성된다.

사용자가 모호하게 요청하면 서비스가 기본값을 제시하거나, 필요한 경우 질문을 통해 조건을 좁힐 수 있다.

### 바이브 데이트 코스 수정

바이브 데이트 코스 수정은 자연어 기반 데이트 코스 생성 및 편집 요청이다.

AI 에이전트가 사용자의 요청을 해석해 추천 결과나 코스를 수정한다.

수정 가능한 기본 단위는 장소와 코스다.

가능한 수정:

- 장소 추가
- 장소 삭제
- 장소 교체
- 코스 순서 변경
- 코스 구조 변경
- 분위기 변경
- 이동 거리 줄이기
- 카테고리 변경

## 9. 화면 단위 요구사항

### 제품 형태

MVP는 모바일 앱 중심으로 설계한다.

데이트 장소를 찾는 상황이 이동 중이거나 휴대폰으로 탐색하는 경우가 많기 때문에, 첫 화면 경험은 모바일에 최적화한다.

### 회원가입 화면

- 이메일, 소셜 로그인 등 최소 정보로 가입한다.
- 복잡한 프로필 입력은 제외한다.

### 로그인 화면

- 기존 사용자가 저장된 성향으로 바로 추천받기 위한 진입점이다.

### 성향 선택 온보딩 화면

- 문장형 성향 선택지를 제공한다.
- 사용자는 제한 없이 여러 개 선택할 수 있다.
- 선택한 성향은 기본 데이트 성향으로 저장된다.

### 지도 기반 추천 화면

- 앱의 메인 화면이다.
- 지도 중심 + 하단 리스트 구조로 구성한다.
- 현재 지도에 보이는 화면 영역을 기준으로 플레이스 추천이 이루어진다.
- 사용자가 지도를 이동하거나 확대/축소하면 해당 지도 영역 기반으로 추천 결과를 갱신한다.

### 플레이스 리스트 / 카드 영역

- 모바일에서는 지도 하단 바텀시트 형태로 표시한다.
- 플레이스는 성향 기반 점수순으로 정렬한다.
- 각 카드에는 간단 요약, 선호 항목별 점수, 외부 서비스 평점, 외부 링크를 표시한다.

### 지도 마커

- 추천 점수에 따라 시각적 강약을 적용한다.
- 점수가 높을수록 진하게 표시한다.
- 점수가 낮을수록 옅게 표시한다.

### 바이브 데이트 코스 수정 채팅창

- 지도 화면 하단에 고정 입력창을 둔다.
- 기본 상태에서는 한 줄 입력창만 보인다.
- 대화가 시작되면 확장형 채팅 바텀시트로 표시한다.
- 사용자는 지도를 보면서 바로 자연어로 데이트 코스 생성 또는 수정 요청을 입력할 수 있다.

### 바이브 데이트 코스 결과 표시 영역

- 사용자가 채팅창에서 요청했을 때 코스를 생성한다.
- 지도 위에 경로 또는 순서를 표시한다.
- 하단 리스트에는 코스 순서대로 장소를 표시한다.
- 이후 자연어로 장소 교체, 삭제, 순서 변경이 가능하다.

### 내 성향 수정 화면

- 가입 시 선택한 성향 문장을 나중에 수정할 수 있다.
- 수정된 성향은 이후 추천에 반영된다.

## 10. 데이터 및 정책 요구사항

### 저장할 플레이스 데이터

MVP에서 저장하는 플레이스 데이터는 Vibe Dating이 직접 관리하는 데이터로 제한한다.

- 카테고리 (우리 정의, [장소-카테고리.md](장소-카테고리.md) 기준)
- 외부 링크
- 외부 서비스 평점
- 간단 요약
- 태그별 점수

장소명, 주소, 좌표 등 네이버 지도 API에서 제공되는 기본 플레이스 정보는 우리 DB에서 직접 관리하지 않고, 네이버 API 응답을 사용한다.

플레이스 식별을 위해 각 플레이스는 네이버 지도 API의 플레이스 ID와 1:1로 매핑한다.

### 외부 서비스 평점

외부 서비스 평점은 여러 출처의 정보를 함께 사용할 수 있다.

출처 예시:

- 네이버
- 카카오
- Google

외부 서비스 평점은 사용자가 추천 결과를 판단하는 데 필요한 정보이므로, MVP에서 계속 보여주는 것을 전제로 한다.

### 외부 링크

외부 링크는 네이버 지도를 1순위로 둔다.

필요에 따라 카카오맵, Google Maps, 공식 홈페이지, 인스타그램 링크를 함께 제공할 수 있다.

### 저장하지 않을 데이터

아래 데이터는 우리 DB에 저장하지 않는다.

- 리뷰 원문
- 블로그 문장 원문
- 외부 플랫폼 사진
- 메뉴판 이미지

Vibe Dating은 외부 콘텐츠 원문을 복제하는 서비스가 아니라, 외부 정보와 자체 분석을 기반으로 데이트 맥락에 맞는 추천을 제공하는 서비스로 정의한다.

### 별도 기술 검토 문서 필요 항목

아래 항목은 MVP 기능 명세서가 아니라 별도 기술 검토 문서에서 다룬다.

- 웹 크롤링 / 웹서치 기반 플레이스 정보 수집 방식
- 외부 API 사용 범위
- 외부 데이터 저장 가능 범위
- AI 기반 태그별 점수 산정 로직
- 사용자 성향 기반 추천 점수 계산 방식
- 선호 항목별 점수 표시 방식
- AI 에이전트 기반 바이브 데이트 코스 수정 처리 방식

## 11. MVP 기능 명세

### 회원가입

목적:

사용자의 기본 데이트 성향을 저장하기 위한 계정을 생성한다.

요구사항:

- 사용자는 최소 정보로 회원가입할 수 있어야 한다.
- 회원가입 이후 성향 선택 온보딩으로 이동해야 한다.
- MVP에서는 커플 계정 구조를 제공하지 않는다.

### 로그인

목적:

기존 사용자가 저장된 기본 성향을 기반으로 추천을 받을 수 있게 한다.

요구사항:

- 사용자는 로그인 후 지도 기반 추천 화면으로 진입할 수 있어야 한다.
- 로그인한 사용자의 기본 성향이 추천에 반영되어야 한다.

### 성향 선택 온보딩

목적:

사용자의 장기 데이트 취향을 수집한다.

요구사항:

- 성향은 문장형 선택지로 제공한다.
- 사용자는 제한 없이 여러 개 선택할 수 있어야 한다.
- 선택한 성향은 사용자 기본 성향으로 저장되어야 한다.

### 플레이스 데이터 수집 및 AI 점수화 어드민

목적:

운영자가 MVP 추천에 사용할 플레이스 데이터를 수집하고, AI 분석을 통해 플레이스별 성향 점수와 추천에 필요한 기초 정보를 생성한다.

요구사항:

- 운영자는 어드민 페이지에서 플레이스를 등록하거나 조회할 수 있어야 한다.
- 운영자는 플레이스별로 웹 크롤링 또는 웹서치를 실행할 수 있어야 한다.
- 어드민은 수집된 출처 링크, 외부 데이터 메타 정보, 요약 정보를 확인할 수 있게 제공해야 한다.
- 어드민은 수집된 정보를 기반으로 AI 분석을 실행할 수 있어야 한다.
- AI 분석 결과로 태그별 점수와 간단 요약을 생성할 수 있어야 한다.
- 운영자는 생성된 점수와 요약 정보를 Place DB에 저장하거나 재생성할 수 있어야 한다.
- 외부 리뷰 원문, 블로그 문장 원문, 외부 플랫폼 사진, 메뉴판 이미지는 저장하지 않는다.

### 지도 기반 플레이스 추천

목적:

현재 지도 화면 영역 안에서 사용자의 성향에 맞는 데이트 플레이스를 추천한다.

요구사항:

- 현재 지도에 보이는 영역을 기준으로 플레이스를 조회한다.
- 조회된 플레이스는 사용자 성향 기반 추천 점수순으로 정렬한다.
- 지도와 리스트를 함께 제공한다.
- 추천 점수에 따라 지도 마커의 시각적 강약을 다르게 표시할 수 있어야 한다.

### 플레이스 카드

목적:

사용자가 추천된 장소를 빠르게 판단할 수 있게 한다.

요구사항:

- 장소명, 카테고리, 외부 서비스 평점, 간단 요약, 선호 항목별 점수를 표시한다.
- 외부 지도/플레이스 링크를 제공한다.
- 외부 링크는 네이버 지도를 우선한다.

### 바이브 데이트 코스 수정 채팅

목적:

사용자가 자연어로 데이트 코스를 생성하거나 수정할 수 있게 한다.

요구사항:

- 지도 화면 하단에 자연어 입력창을 제공한다.
- 사용자는 추천된 플레이스 또는 코스에 대한 생성, 수정, 재생성 요청을 입력할 수 있다.
- 시스템은 요청을 해석해 추천 플레이스 목록을 조정하거나 데이트 코스를 생성 또는 수정한다.
- 모호한 요청은 기본값으로 처리하거나, 필요한 경우 사용자에게 추가 질문을 할 수 있다.

### 바이브 데이트 코스 수정: 코스 생성

목적:

사용자가 요청했을 때 추천된 플레이스들을 조합해 데이트 코스를 만든다.

요구사항:

- 코스는 사용자가 요청했을 때만 생성한다.
- 코스는 여러 플레이스의 순서 있는 조합으로 표현한다.
- 지도 위에 순서 또는 경로를 표시한다.
- 하단 리스트에는 코스 순서대로 장소를 표시한다.

### 바이브 데이트 코스 수정: 코스 편집

목적:

생성된 코스를 사용자가 원하는 분위기와 조건에 맞게 대화형으로 수정한다.

요구사항:

- 장소 추가, 삭제, 교체가 가능해야 한다.
- 코스 순서 변경이 가능해야 한다.
- 카테고리 변경, 분위기 변경, 이동 거리 축소 같은 자연어 요청을 처리할 수 있어야 한다.
