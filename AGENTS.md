# AGENTS.md

## 프로젝트 개요

Vibe Dating은 커플이 직접 네이버 지도, 블로그, 검색 결과를 오가며 데이트 장소를 찾는 수고를 줄여주는 데이트 장소 추천 서비스다.

핵심은 일반적인 소개팅/매칭 앱이 아니라, 커플의 성향과 오늘의 무드에 맞는 장소 후보를 압축해서 보여주는 큐레이션 레이어를 만드는 것이다.

한 줄로 정의하면:

> Vibe Dating은 사용자의 데이트 성향을 기반으로 현재 지도 영역 안의 데이트 장소를 추천하고, 자연어로 장소와 코스를 수정할 수 있게 해주는 지도 기반 데이트 큐레이션 서비스다.

## 서비스가 해결하려는 문제

사람들은 주말 데이트 전날이나 데이트 계획을 짤 때 네이버 지도에서 장소를 검색하고, 플레이스 페이지를 하나씩 열어보며 괜찮은 곳을 직접 고른다.

이 방식은 장소 정보는 풍부하지만, 사용자가 직접 많은 후보를 비교해야 한다. 즉 문제는 장소 정보가 없다는 것이 아니라, 내 취향에 맞는 괜찮은 후보를 찾기까지 반복 탐색과 선별 공수가 많이 든다는 점이다.

Vibe Dating이 줄여주려는 핵심 수고는 다음과 같다.

- 데이트 장소를 일일이 검색하는 수고
- 플레이스 페이지를 하나씩 열어 비교하는 수고
- 내 성향에 맞는지 직접 판단하는 수고
- 여러 장소를 조합해 코스로 만드는 수고

## 제품 포지셔닝

Vibe Dating은 네이버 지도, 카카오맵, Google Maps를 대체하려는 서비스가 아니다.

외부 지도 서비스는 플레이스의 상세 정보를 제공하는 서비스이고, Vibe Dating은 그 위에서 사용자의 성향에 맞는 장소와 코스를 골라주는 추천/편집 레이어다.

구체적인 플레이스 정보 확인은 네이버, 카카오, Google 같은 외부 링크로 연결한다. Vibe Dating은 사용자가 어떤 장소를 볼지, 어떤 조합을 고려할지 먼저 좁혀주는 역할을 한다.

## 핵심 사용자 경험

초기 핵심 경험은 다음 흐름이다.

1. 사용자가 회원가입 시 기본 데이트 성향을 문장형 선택지로 고른다.
2. 사용자가 지도 화면에 들어간다.
3. 현재 지도에 보이는 영역 안의 데이트 가능 플레이스가 조회된다.
4. 저장된 사용자 성향을 기준으로 플레이스가 추천 점수순으로 정렬된다.
5. 지도 마커와 하단 리스트에서 추천 결과를 함께 본다.
6. 사용자는 채팅창에 자연어로 `조금 더 조용한 곳으로 바꿔줘`, `여기서 코스 하나 만들어줘`, `저녁 파스타 말고 고기`처럼 요청한다.
7. 서비스는 추천 결과나 코스를 대화형으로 수정한다.

## 핵심 가치

- 커플 취향에 맞는 데이트 장소 후보를 빠르게 좁혀준다.
- 장소를 단순 나열하지 않고, "어떤 커플에게, 어떤 상황에서, 어떤 무드로 좋은가"를 중심으로 추천한다.
- 장기적으로는 장소별 호/불호 피드백을 누적해 추천 품질을 점진적으로 개선한다.
- 외부 지도/검색 데이터가 아니라 자체적으로 정리한 데이트 맥락 데이터와 Place DB를 핵심 자산으로 본다.
- 사용자가 추천 결과를 자연어로 수정할 수 있는 `바이브 수정` 경험을 차별점으로 본다.

## MVP 방향

- MVP는 모바일 앱 중심으로 설계한다.
- 초기에는 모든 지역을 열지 않고, 유명 데이트 상권/핫플 지역 일부부터 제한 오픈한다.
- 회원가입/로그인을 MVP에 포함한다. 가입 시 사용자의 기본 데이트 성향을 문장형 선택지로 저장한다.
- 성향 문장은 `조용한 곳이 좋아요`, `사진 찍는 걸 좋아해요`처럼 사용자 언어에 가까운 문장형 선택지로 제공하고, 사용자는 제한 없이 여러 개 선택할 수 있다.
- MVP 메인 경험은 현재 지도 화면 영역 안의 데이트 가능 플레이스를 조회하고, 저장된 사용자 성향 기반 추천 점수순으로 지도와 하단 리스트에 보여주는 것이다.
- 지도 마커는 추천 점수에 따라 시각적 강약을 다르게 표현한다. 높은 점수는 진하게, 낮은 점수는 옅게 표현하는 방향을 우선 고려한다.
- 플레이스 카드에는 간단 요약, 추천 이유, 평점, 외부 링크를 제공한다. 외부 링크는 네이버 지도를 1순위로 둔다.
- MVP의 핵심 검증 대상은 `성향 기반 장소 추출이 잘 되는지`, `추천이 실제로 유용한지`, `대화형 수정이 좋은 사용자 경험을 만드는지`다.
- MVP의 기본 출발점은 장소 추천이다. 코스는 자동으로 항상 생성하지 않고, 사용자가 채팅창에서 요청했을 때 추천된 플레이스나 선택된 플레이스를 조합해 생성한다.
- `바이브 수정`은 MVP 핵심 기능이다. 사용자는 자연어로 추천 결과와 코스를 추가, 삭제, 교체, 재정렬, 분위기 변경, 카테고리 변경, 이동 거리 축소 등으로 수정할 수 있어야 한다.
- 좋아요/싫어요 피드백은 장기적으로 중요하지만 1차 개발 범위에서는 후순위로 둔다.
- 예약, 결제, 커플 계정, 상세 리뷰 작성, 업주용 장소 등록, 운영자 검수 어드민 고도화는 MVP에서 제외한다.

## 도메인 원칙

- MVP에서 성향은 개인 계정에 저장되는 기본 데이트 성향으로 다룬다. 장기적으로는 커플 맥락을 표현해야 하지만, MVP에서는 커플 계정 없이 데이트 계획을 주도하는 사용자의 성향 프로필로 출발한다.
- 태그는 장소 검색 필터가 아니라 추천 스코어링과 추천 이유를 구성하는 도메인 언어다. 사용자에게는 문장형 성향으로 보이고, 내부적으로는 추천 계산용 태그로 매핑될 수 있다.
- 추천 점수는 플레이스의 절대 점수가 아니다. 같은 장소라도 사용자가 선택한 성향에 따라 상대적으로 다른 점수를 가져야 한다.
- Place DB에는 장소명, 주소, 좌표, 카테고리, 외부 링크, 평점, 간단 요약, 태그별 점수 같은 자체 맥락 데이터를 저장한다.
- 플레이스별 태그 점수는 웹서치/외부 데이터/자체 분석 정보를 바탕으로 AI가 산정할 수 있으며, 구체적인 점수 산정 방식은 기술 검토 문서에서 다룬다.
- 추천 이유는 사용자의 성향, 플레이스의 태그별 점수, 장소 정보, 코스 맥락을 기반으로 생성한다. 확인되지 않은 사실을 단정하거나 외부 원문을 재사용하지 않는다.
- 외부 API와 검색 결과는 후보 발굴, 현재성 확인, 링크 연결, 교차 검증에 우선 사용한다.
- 외부 리뷰 원문, 블로그 문장 원문, 외부 플랫폼 사진, 메뉴판 이미지는 저장하지 않는다.
- 평점은 사용자 판단에 필요한 정보이므로 네이버, 카카오, Google 등 여러 출처를 활용할 수 있다. 단, 출처별 저장 가능 범위와 표시 조건은 반드시 확인한다.
- 약관, robots.txt, 저작권, 저장 가능 범위는 데이터 수집 설계에서 항상 확인한다.

## 기술 방향

- 기술 검토는 `docs/mvp-technical-review.md`를 기준으로 한다.
- 핵심 기술 흐름은 `Place DB -> AI 태깅 -> 사용자 성향 기반 추천 점수 -> 지도 기반 플레이스 조회 -> 바이브 수정 에이전트`다.
- 지도 조회는 현재 지도 화면 영역(viewport/bounding box)을 기준으로 한다.
- 지도 이동/확대/축소에 따른 조회는 debounce, 캐싱, 좌표 인덱싱을 고려한다.
- 플레이스 태그 점수, 요약, 기본 추천 이유처럼 사전 계산 가능한 값은 최대한 사전 생성하고, 실시간 LLM 호출은 바이브 수정과 코스 생성 중심으로 제한한다.
- 바이브 수정 에이전트는 자연어 요청을 구조화된 action으로 해석하는 방향을 우선 검토한다.
- 코스 생성은 사용자 요청 기반이며, MVP에서는 정교한 경로 최적화보다 추천 플레이스 조합과 순서 구성에 집중한다.

- 프로젝트 관련 질문에는 먼저 `docs/` 문서를 기준으로 답한다.
- MVP 기능 범위는 우선 `docs/mvp-feature-spec.md`를 기준으로 답한다.
- MVP 논의 과정이나 결정 배경이 필요하면 `docs/mvp-feature-spec-steps/`의 단계별 문서를 확인한다.
- 추천, AI 태깅, 지도 조회, 외부 데이터 수집, 바이브 수정 에이전트처럼 구현 방식이 중요한 질문은 `docs/mvp-technical-review.md`를 기준으로 답한다.
- 문서에 없는 내용은 확정된 사실처럼 말하지 않고, 가설이나 제안으로 구분한다.
- 새로운 제품/도메인 결정을 내리면 관련 문서에 반영하는 것을 우선 고려한다.
- 커플 유형, 문장형 성향, 내부 태그, 장소 태그, 상황 조건, 추천 기준, Place DB 스키마를 논의할 때는 각 개념을 혼동하지 않는다.
- 기능 명세서에는 사용자 경험과 제품 범위를 쓰고, 추천 공식/AI 태깅/크롤링/에이전트 처리 같은 구현 로직은 기술 검토 문서로 분리한다.

## MVP 검증 기준

- MVP의 핵심 성공 기준은 한 번 추천을 사용한 사용자가 이후 다시 방문해 데이트 장소 추천 또는 바이브 수정 기능을 재사용하는 것이다.
- 함께 볼 지표는 회원가입 완료율, 성향 선택 완료율, 추천 결과 클릭률, 외부 링크 클릭률, 바이브 수정 사용률, 코스 생성 요청률, 재방문율, 재방문 후 기능 재사용률, 사용자 만족도 조사 결과다.
- MVP는 지인 또는 소규모 사용자 테스트로 먼저 검증하고, 추천 품질, 지도 기반 탐색 경험, 바이브 수정 경험을 확인한 뒤 공개 범위를 확장한다.

## 세션 마무리 트리거

사용자가 `"세션 마무리"`라고 말하면, 이번 대화에서 장기적이고 글로벌한 지침으로 남길 만한 후보만 정리해서 사용자에게 보낸다.

- 형식: 키워드와 핵심 중심의 간결한 개괄식
- 포함: 반복 가능한 작업 방식, 장기 선호, 프로젝트 밖에서도 유효한 지침
- 제외: 일회성 작업 내용, 임시 맥락, 특정 버그의 세부 해결 과정
- 저장: 승인 전에는 지침에 반영하지 않고, 사용자가 승인한 항목만 따로 저장 대상으로 다룬다.
